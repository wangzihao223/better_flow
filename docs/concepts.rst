核心概念
========

设计思想
--------

``node-flow`` 的核心思想是把一个流程拆成“节点 + 事件 + 图 + 运行时”。
业务代码专注于“输入是什么、输出是什么”，调度、并发、分发和观察交给
runtime。

这种拆法的好处是：

- 节点足够小，职责边界清楚
- 分支和并行是图结构的自然结果，不需要写大量控制流代码
- 同一条流程可以混合同步、异步和 CPU 任务
- runtime 可以统一做统计、错误记录和 hook 观察

它不是用来替代所有编排系统的；更准确地说，它是一个“嵌入式流程编排
内核”，适合在 Python 应用内部组织流程。

Event
-----

``Event`` 是节点之间传递的数据包，包含事件 ID、来源节点、事件名、业务
payload、创建时间、trace 和路由目标。

Node
----

节点是业务执行单元。节点读取 ``event.payload``，返回新的处理结果。
runtime 根据节点的 ``execution_mode`` 选择线程池、asyncio loop 或进程池。

节点的职责很窄：

- 不关心整个流程怎么被调度
- 不直接管理线程、进程或事件循环
- 不需要知道自己是被 trigger 还是被下游传播触发

这让节点更容易测试，也更容易复用。

Graph
-----

``WorkflowGraph`` 只维护节点和连接关系，不负责执行。它提供节点注册、
上下游查询和重复 ``node_id`` 检查。

图结构的价值在于：分支、路由和并行都由连接关系表达，而不是靠分散在
各个节点里的条件判断来控制。

Runtime
-------

``WorkflowRuntime`` 负责节点注册、事件分发、线程池、进程池、共享 asyncio
loop、timer 和 runtime hook。

它的架构优势主要有三点：

1. **统一调度**

   同一套 runtime 同时管理同步、异步和 CPU 任务，不需要三套执行系统。

2. **统一观察**

   hook 可以观察事件创建、任务提交、节点执行、错误和运行状态，便于
   日志、调试和后续监控。

3. **统一边界**

   节点只负责业务，图只负责拓扑，runtime 只负责执行和生命周期。这个分
   层让代码更容易理解，也更容易继续扩展。

返回值协议
----------

节点返回值由 runtime 统一处理：

- ``None``：停止传播
- 普通值：替换 ``event.payload`` 后继续传播
- ``Event``：使用完整事件继续传播
- ``list``：拆成多个事件继续传播

路由语义
--------

``RouterNode`` 通过 ``event.route_targets`` 控制当前一跳的目标：

- ``None``：广播给所有下游
- ``[]``：不传播
- ``["a", "b"]``：只发给指定下游节点

适用边界
--------

它适合：

- 事件驱动的业务流程
- 简单到中等复杂度的并行编排
- 需要把执行资源统一到一个 runtime 里管理的应用

它暂时不适合：

- 需要复杂持久化编排、重试策略、分布式调度的大型工作流平台
- 需要严格的任务持久化、断点恢复、跨进程协调能力的系统
